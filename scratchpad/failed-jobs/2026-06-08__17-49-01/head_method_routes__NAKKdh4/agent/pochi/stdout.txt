# Explicit `HEAD` / `OPTIONS` / 405 Handling with RedwoodSDK ## Background RedwoodSDK's router does **not** automatically map `HEAD` requests to a `GET` handler — they have to be wired up explicitly, otherwise the route returns `405 Method Not Allowed`. Build a small in‑memory file metadata API on top of RedwoodSDK that correctly handles `GET`, `HEAD`, `OPTIONS`, unknown ids, and disallowed methods, against the **real** `npm run dev` server. ## Requirements * Use the pre-scaffolded RedwoodSDK project at the project path below (Vite plugin + `defineApp` + `rwsdk/router`). * Implement a single endpoint `/api/files/:id` that supports `GET`, `HEAD`, and `OPTIONS`. * Seed an in-memory file metadata store with exactly three entries (ids `f1`, `f2`, `f3` — exact name/size/contentType values given below in *Acceptance Criteria*). * `GET /api/files/:id` returns the file metadata as JSON. * `HEAD /api/files/:id` returns the **same** status code and response headers as the corresponding `GET`, but with an **empty body**. It must NOT return `405`. * `OPTIONS /api/files/:id` returns `204 No Content` with the `Allow` header listing the supported methods (`GET, HEAD, OPTIONS`). * Unknown ids (e.g. `/api/files/does-not-exist`) return `404` for `GET`, `HEAD`, and `OPTIONS`. The `HEAD` 404 response must still have an empty body. * Any non-allowed method (`POST`, `PUT`, `DELETE`, `PATCH`) on `/api/files/:id` returns `405 Method Not Allowed` with an `Allow: GET, HEAD, OPTIONS` header. * Implementation runs against `npm run dev` (no mocking, no production build). ## Implementation Hints * RedwoodSDK supports per-method route objects, e.g. `route("/api/files/:id", { get: handler, head: handler, options: handler })` — see `rwsdk/router`'s method-handlers documentation. * `HEAD` handlers must be defined explicitly; you cannot rely on `GET` being reused automatically. * The same handler function can be reused for `GET` and `HEAD` if it builds the headers up first and only writes the body for `GET` (you can inspect `request.method`). * For `HEAD`, the response body must be empty but `Content-Length` must equal the byte length of what `GET` would have returned for the same id. * The automatic OPTIONS / 405 behavior of `rwsdk` may help, but you are responsible for matching the **exact** required status codes and headers described in *Acceptance Criteria* — including 404 on unknown ids for `OPTIONS`. * Read `params.id` from the route handler argument. * Use the standard `Response` constructor;

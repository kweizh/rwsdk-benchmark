## Multi-Room Chat Implementation

I've extended the RedwoodSDK starter project with a multi-room realtime chat application. Here's a summary of the changes:

### Files Modified

**`wrangler.jsonc`** — Added Durable Object bindings and migrations for `SyncedStateServer`:
- `durable_objects.bindings` with `SYNCED_STATE` bound to `SyncedStateServer`
- `migrations` with `new_sqlite_classes: ["SyncedStateServer"]` (tag `v1`)

**`src/worker.tsx`** — Wired up the synced state infrastructure and the chat route:
- Exports `SyncedStateServer` (required by the SDK for DO wiring)
- Registers `syncedStateRoutes((env) => env.SYNCED_STATE)` to handle WebSocket RPC traffic at `/__synced-state` and `/__synced-state/:id`
- Adds `route("/chat/:roomId", Chat)` to render the chat UI

### Files Created

**`src/app/pages/chat.tsx`** — The chat room page component:
- Marked `"use client"` since it uses browser-only hooks
- Extracts `roomId` from `params` (provided by the router from the URL segment)
- Uses `useSyncedState<string[]>([], "messages", roomId)` — the `roomId` third argument scopes state to a specific room, ensuring different rooms have isolated message histories
- Renders:
  - A text `<input>` with `name="message"` and `aria-label="message"` for the verifier to locate
  - A `<button>` with visible text "Send" to submit messages
  - A `<ul>` displaying all submitted messages in order

### How it works

1. **Room isolation**: `useSyncedState` with the `roomId` parameter causes the client to connect to `/__synced-state/<roomId>` instead of `/__synced-state`. The `syncedStateRoutes` uses the `:id` URL parameter to create a separate Durable Object instance per room, giving each room its own isolated state.

2. **Message persistence**: The `SyncedStateServer` Durable Object stores state in memory (backed by SQLite in production). Within a dev server lifetime, messages survive page reloads because the DO instance persists.

3. **Realtime sync**: Messages are synced across all clients viewing the same room via WebSocket-based RPC — when one client calls `setState`, all subscribers to the same key get notified.

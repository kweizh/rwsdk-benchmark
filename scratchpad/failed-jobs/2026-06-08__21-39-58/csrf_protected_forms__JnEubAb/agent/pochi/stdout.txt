Build a small comments API protected by a real CSRF (Cross-Site Request Forgery) defense using rwsdk's request pipeline (global middleware and/or per-route interrupters). The API serves a public read endpoint and a write endpoint that requires a per-session CSRF token, plus an unauthenticated health endpoint that is always reachable. ## Requirements * The web app is a RedwoodSDK project served by `npm run dev` (Vite + Cloudflare workerd). * Mount a CSRF guard in rwsdk's request pipeline so that for any non-safe HTTP method (POST, PUT, DELETE, PATCH) the request is rejected unless an `X-CSRF-Token` header (or a `_csrf` form field) matches a token stored in a `csrf` cookie on the same request. * Provide a token issuance endpoint that mints fresh CSRF tokens, sets them in a cookie, and returns them in JSON so a client can echo them back in subsequent unsafe requests. * Provide a protected write endpoint for posting comments and a public read endpoint for listing them. Storage may be in-memory (per-process). * Provide a health endpoint that uses a safe method and is exempt from CSRF protection. ## Implementation Hints * See the RedwoodSDK docs at https://docs.rwsdk.com for `defineApp`, `route`, global middleware, and interrupters. Middleware functions are placed at the top of the `defineApp` array; interrupters are placed before the final handler in a `route(path, [interrupter, handler])` array. * A CSRF token of 32 hex characters can be produced from 16 bytes of `crypto.getRandomValues(new Uint8Array(16))`. * The CSRF cookie and the submitted token must be compared as strings; reject the request when either is missing or they do not match. Use a constant-time comparison if possible. * Cookies are read from the `Cookie` request header and set with `Set-Cookie` on the response. The framework's request object is a standard `Request`. * Keep storage trivial (a module-level array).

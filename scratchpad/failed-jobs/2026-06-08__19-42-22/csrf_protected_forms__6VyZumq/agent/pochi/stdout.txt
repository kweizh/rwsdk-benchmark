# CSRF-Protected Comments API with RedwoodSDK ## Background You are extending a freshly scaffolded RedwoodSDK (`rwsdk`) project. Build a small comments API protected by a real CSRF (Cross-Site Request Forgery) defense using rwsdk's request pipeline (global middleware and/or per-route interrupters). The API serves a public read endpoint and a write endpoint that requires a per-session CSRF token, plus an unauthenticated health endpoint that is always reachable. ## Requirements * The web app is a RedwoodSDK project served by `npm run dev` (Vite + Cloudflare workerd). * Mount a CSRF guard in rwsdk's request pipeline so that for any non-safe HTTP method (POST, PUT, DELETE, PATCH) the request is rejected unless an `X-CSRF-Token` header (or a `_csrf` form field) matches a token stored in a `csrf` cookie on the same request. * Provide a token issuance endpoint that mints fresh CSRF tokens, sets them in a cookie, and returns them in JSON so a client can echo them back in subsequent unsafe requests. * Provide a protected write endpoint for posting comments and a public read endpoint for listing them. Storage may be in-memory (per-process). * Provide a health endpoint that uses a safe method and is exempt from CSRF protection. ## Implementation Hints * See the RedwoodSDK docs at https://docs.rwsdk.com for `defineApp`, `route`, global middleware, and interrupters. Middleware functions are placed at the top of the `defineApp` array; interrupters are placed before the final handler in a `route(path, [interrupter, handler])` array. * A CSRF token of 32 hex characters can be produced from 16 bytes of `crypto.getRandomValues(new Uint8Array(16))`. * The CSRF cookie and the submitted token must be compared as strings; reject the request when either is missing or they do not match. Use a constant-time comparison if possible. * Cookies are read from the `Cookie` request header and set with `Set-Cookie` on the response. The framework's request object is a standard `Request`. * Keep storage trivial (a module-level array). This is an evaluation app, not production. ## Acceptance Criteria * Project path: /home/user/myproject * Start command: npm run dev * Port: 5173 * Token endpoint (always reachable, no CSRF check): * `GET /csrf` -> HTTP `200`, `Content-Type: application/json`. * Response body JSON shape: `{"token": string}` where `token` matches the regex `^[0-9a-f]{32}$` (32 lowercase hex characters). * Response MUST include a `Set-Cookie` header named `csrf` whose value equals the JSON `token` value, with attributes that include `SameSite=Strict` (case-insensitive) and `Path=/`. * Each request to `GET /csrf` MUST return a freshly generated token (two consecutive calls return different tokens). * Health endpoint (always reachable, no CSRF check): * `GET /api/health` -> HTTP `200`, `Content-Type: application/json`, response body JSON `{"status": "ok"}`. * Protected write endpoint: * `POST /api/comments` with `Content-Type: application/json`, body `{"author": string, "text": string}`: * When the request carries a `csrf` cookie AND an `X-CSRF-Token` request header whose value equals that cookie value -> HTTP `201`, `Content-Type: application/json`, response body JSON `{"id": number, "author": string, "text": string}` echoing back the submitted `author` and `text`. `id` MUST be a positive integer that is unique within the process. * When `Content-Type: application/x-www-form-urlencoded` is used and the form body contains a `_csrf` field whose value equals the `csrf` cookie value -> the request MUST be accepted with the same `201` response shape (using the `author` and `text` form fields). * When the `X-CSRF-Token` header is missing AND no `_csrf` form field is present (with or without the cookie) -> HTTP `403`, `Content-Type: application/json`, response body JSON exactly `{"error": "invalid_csrf_token"}`. No new comment may be stored. * When the `X-CSRF-Token` header is present but its value does NOT match the `csrf` cookie value (or the cookie is missing) -> HTTP `403`, `Content-Type: application/json`, response body JSON exactly `{"error": "invalid_csrf_token"}`. No new comment may be stored. * Public read endpoint: * `GET /api/comments` -> HTTP `200`, `Content-Type: application/json`, response body JSON array of comment objects in the shape `[{"id": number, "author": string, "text": string}, ...]`. The list MUST include every comment previously accepted by `POST /api/comments` in this process. * General behavior: * The CSRF check MUST only apply to non-safe HTTP methods (POST, PUT, DELETE, PATCH). Safe methods (GET, HEAD) MUST never be rejected because of CSRF. * The CSRF token MUST be generated dynamically per `GET /csrf` request; the implementation MUST NOT hardcode a constant token value.
